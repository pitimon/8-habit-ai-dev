# Skill Effectiveness Report

**Status**: Maintainer-curated trend tracker (v2.6.1) | **Updated by**: Plugin maintainer | **Data source**: `~/.claude/lessons/*.md` (`## Skill effectiveness` Q6) **+ invocation ground truth** from `~/.claude/projects/*` transcripts | **Related**: [Issue #92](https://github.com/pitimon/8-habit-ai-dev/issues/92), [Issue #314](https://github.com/pitimon/8-habit-ai-dev/issues/314), [`skills/reflect/SKILL.md`](skills/reflect/SKILL.md) Q6

> **Governing principle (how this tally may be used).** Usage history is a signal for **discoverability** (is the right skill reachable when its moment comes?) and **footprint** (does a rarely-relevant skill justify its line in the every-session banner?) — it is **never a trigger for revocation**. Revocation requires a stronger criterion: (1) the skill is superseded/duplicated across the [plugin boundary](CLAUDE.md), (2) it causes demonstrated harm with a documented root cause, or (3) the need it addresses no longer exists. **Low usage is none of these.** Rationale: the Q6-only signal under-counted real invocation for 5 of the 14 "zero-signal" skills (see #314 — `requirements` alone was invoked 14×), and frequency measures _demand realized so far_, not _value_ — fire-extinguisher skills (`/post-mortem`, `/diagnose`, `/security-check`, `/eu-ai-act-check`, `/calibrate`) are low-frequency by design and high-value when they fire.

## Purpose

Systematic tracking of which 8-habit skills users find **most useful**, which they find **least useful or confusing**, and which skills **missed a moment where they should have triggered** over time. This is **H7 (Sharpen the Saw) applied to the plugin itself** — invest in the capability of the skills, not just the output they produce.

Without a signal like this, skills drift: some become verbose padding, some get forgotten, some quietly confuse users who never say so, and some fail to activate when the situation needs them. `/reflect` Q6 is the cheapest possible signal — one skill name per field, or "n/a".

## How the data flows

```
User runs a task → invokes /reflect → answers Q6 (most_useful / least_or_confusing / missed_skill)
  ↓
/reflect writes ~/.claude/lessons/YYYY-MM-DD-<slug>.md with `## Skill effectiveness` section
  ↓
(user-local — lives in user home, not plugin repo, not shared automatically)
  ↓
Periodically (monthly / per release), maintainer reads their own lessons
  (and contributor reports if shared) and updates this file
  ↓
Trend analysis informs /skill-improve cycles, discoverability + footprint decisions, and scope (never skill removal — see Governing principle)
```

## What this file is NOT

- ❌ **Not auto-generated** — skills are read-only guidance, not runtime aggregators. Cross-file lesson scanning would need either a runtime component (belongs in `claude-governance` per plugin boundary) or a manual maintainer step. This doc chooses manual.
- ❌ **Not enforcement** — low scores do not block skill invocation or trigger automated deprecation. Humans read, humans decide.
- ❌ **Not a ranking** — "least useful in context A" can be "most useful in context B". Trend notes capture nuance the raw count cannot.
- ❌ **Not a leaderboard** — the goal is improvement signal, not competition between skills.

## Maintainer update protocol

When updating this file:

1. **Read lesson files**: `ls ~/.claude/lessons/*.md` on the maintainer's machine, plus any contributor reports shared via PR or issue.
2. **Extract Q6 data**: `grep -A 4 '## Skill effectiveness' ~/.claude/lessons/*.md` — harvest legacy "Most useful" / "Least useful / confusing" lines and current `most_useful`, `least_or_confusing`, and `missed_skill` fields.
3. **Harvest invocation ground truth** (added per #314 — Q6 is blind to Skill-tool, script, and cross-project runs): count real invocations across all projects, not just reflections:
   - User slash invocations: `grep -rhoE '<command-name>/8-habit-ai-dev:[a-z-]+</command-name>' ~/.claude/projects/ | sort | uniq -c`
   - Claude Skill-tool invocations: `grep -rhoE '8-habit-ai-dev:[a-z-]+' ~/.claude/projects/` filtered to `"name":"Skill"` call sites.
   - **Discount rules — apply or the count inverts the error**: (a) **exclude `claude-mem-observer-sessions`** (the observer pipeline auto-runs skill scripts — it inflated `/ai-dev-log` to 91 of which 69 were observer noise); (b) **exclude the plugin's own repo dir** `-Users-itarun-8-habit-ai-dev` (dogfooding/self-test ≠ adoption); (c) **ignore "transcripts-mentioning" totals** — a skill name appears in every session via the SessionStart banner (~224 uniform floor), which is presence, not usage.
4. **Tally + classify**: increment per-skill counters; set each skill's **Class** (`invoked` / `delegated` / `expected-zero`) in the Skill class index below from the invocation evidence. Skip `n/a` entries.
5. **Update `Last updated`**, `Lessons analyzed`, and any **trend notes** per skill.
6. **Flag for action — discoverability/footprint only** (per the Governing principle above): if a skill shows an active _domain_ served ad-hoc (missed-trigger signal, or cross-project work done by other means), route it to `/skill-improve` as a **discoverability** question, or demote it from the session-start banner as a **footprint** decision. **Never** flag low usage as a deprecation candidate — revocation needs supersession, demonstrated harm, or obsolescence.
7. **Commit** the updated report alongside other maintenance in a `chore(skill-eff): update report` commit.
